# Rust & Logic 设计系统

本文档用于梳理当前 Web UI 的设计系统，目标不是冻结视觉稿，而是把已经稳定下来的设计语言、布局规则、组件语义和扩展边界整理出来，便于后续继续迭代、复用，或迁移到桌面壳层。

## 1. 适用范围

这套设计系统当前服务于 `Rust & Logic` 的游戏层，不服务于 `TapeScript` 运行时本体。

- `TapeScript`
  - 负责语言、编译、VM、硬件边界。
  - 不依赖游戏 UI 风格、剧情演出、资源面板或存档表现。
- `Rust & Logic`
  - 负责玩家可见的界面、地图演出、任务流、资源展示、剧情对话、调试入口。
  - 当前这套设计系统只约束这一层。

因此，后续如果要做编辑器、桌面壳、研发工具或图形编辑器，应优先继承这里的视觉语言与组件语义，但不要反向让 UI 规则侵入 `TapeScript` 核心包。

## 2. 设计目标

### 2.1 气质关键词

- 工业终端
- 克制
- 冷静
- 低饱和废土
- 轻微 CRT / glitch
- 高信息密度
- 玩家界面安静，开发信息降权

### 2.2 设计原则

1. 玩家主路径优先  
   地图、代码、任务、资源是主路径；差异、日志、诊断是辅助路径。

2. 调试信息与玩家信息分层  
   编译诊断、运行时中断、DevLog 都存在，但默认不抢主舞台。

3. 真实地图始终优先  
   地图从启动开始就是真实地图；剧情只是覆盖在真实地图之上的演出层。

4. 语言、数据、表现分离  
   玩家文本来自 `apps/web/i18n.csv`；结构数据来自 `apps/web/app-data.json`；样式和组件语义来自 `apps/web/styles.css`。

5. 终端感不是复古堆砌  
   保留扫描线、彩虹条、轻微辉光和大写标签，但避免厚重拟物、过多装饰和廉价噪声。

## 3. 设计令牌

当前令牌定义在 [apps/web/styles.css](/D:/LabProjects/RustAndLogic/apps/web/styles.css) 的 `:root`。

### 3.1 字体

- `--font-mono`
  - 用于代码、行号、日志、数值、终端字段。
  - 当前别名为 `RAL Mono`，优先映射本机可用等宽字体。
- `--font-body`
  - 用于正文、说明、剧情文本。
  - 当前别名为 `RAL Body`。
- `--font-display`
  - 用于标题、标签、面板抬头、状态强调。
  - 当前别名为 `RAL Display`。

建议继续维持“三层字体分工”，不要把所有区域重新压回单一字体。

### 3.2 色板

核心颜色：

- 背景
  - `--bg: #060302`
  - `--bg-2: #0b0503`
  - `--bg-deep: #0d0805`
- 面板
  - `--panel-surface: #1a120b`
  - `--panel: rgba(15, 8, 5, 0.94)`
  - `--panel-soft: rgba(19, 10, 6, 0.86)`
- 文本
  - `--text: #f0d8bb`
  - `--text-strong: #ffd8ad`
  - `--muted: #664d3d`
- 边界
  - `--border-color: #33261c`
  - `--line: rgba(196, 116, 41, 0.38)`
  - `--line-soft: rgba(196, 116, 41, 0.18)`
- 强调
  - `--accent: #f28d35`
  - `--accent-soft: #a64f21`
  - `--accent-hot: #ffb163`
  - `--accent-glow: rgba(242, 141, 53, 0.4)`
- 状态
  - `--danger: #db5a42`
  - `--danger-bg: rgba(219, 90, 66, 0.16)`
  - `--success: #ffb96c`

规则：

- 主界面以深褐黑为底，琥珀色只做焦点和反馈，不做整面铺色。
- `muted` 文字承担结构信息，不承担主叙事。
- 危险色只用于错误、阻塞、中断，不参与普通强调。

### 3.3 网格与世界

- 世界格尺寸：`40px`
- 机器人尺寸：`24px`
- 资源点尺寸：`22px`
- 基地尺寸：`22px`
- 障碍尺寸：`36px`

这些值当前定义在 [apps/web/src/main.js](/D:/LabProjects/RustAndLogic/apps/web/src/main.js)：

- `WORLD_CELL_SIZE`
- `ROBOT_WORLD_SIZE`
- `DEPOSIT_WORLD_SIZE`
- `BASE_WORLD_SIZE`
- `OBSTACLE_WORLD_SIZE`

后续若继续扩展地图表现，优先从这一组常量出发，避免在 CSS 和脚本里分散硬编码。

### 3.4 间距

当前系统的基础节奏接近 `4 / 8 / 12`：

- 3px: 顶部彩虹条
- 4px: 小型间隙、页码点、进度条高
- 8px: 控制条、按钮内边距、小组件节奏
- 12px: 面板内边距、任务项缩进、正文安全边距

建议未来的新组件优先从这一组节奏生长，不额外发明 5、7、11 这类孤立值，除非是像素还原需要。

### 3.5 边框、辉光、遮罩

- 边框以 `1px` 为主
- 强调辉光统一使用 `--accent-glow`
- 全局扫描线：
  - `body::before`
  - `background-size: 100% 4px`
- 顶部彩虹条：
  - `.app-shell::before`
  - 3px 高
  - 带 `brightness / contrast` 增强
- 位置图遮罩：
  - `.location-art::before`
  - 自上而下融入面板底色

原则是“同一套后效反复复用”，而不是每个面板都单独发明滤镜。

### 3.6 动效

稳定动效口径：

- 面板折叠 / 展开：`0.3s cubic-bezier(0.4, 0, 0.2, 1)`
- 地图运镜 / 角色位移：`cubic-bezier(0.22, 1, 0.36, 1)`
- 进度条宽度：`0.3s`
- 剧情层出现方式：覆盖，不切断底层地图

动效原则：

- 逻辑可以瞬时完成；
- 表现层负责缓动；
- 中断时优先停机，不强行继续播完。

## 4. 布局系统

### 4.1 桌面主布局

当前主壳层是四栏结构，定义在 `.app-shell`：

1. 左侧地点 / 任务栏
2. 代码编辑栏
3. 地图演出栏
4. 右侧系统栏

默认列宽：

- `224px`
- `minmax(348px, 404px)`
- `minmax(640px, 1fr)`
- `236px`

这套布局体现的不是“卡片堆叠”，而是“控制台 + 演出舞台”的工作台结构。

### 4.2 可折叠侧栏

左栏和右栏都支持折叠，折叠后压缩到 `30px`，核心目的：

- 左栏折叠后，让代码区扩展过去
- 右栏折叠后，让地图演出区获得更多空间

当前通过 `body` 数据属性驱动：

- `data-objectives-collapsed`
- `data-right-collapsed`

这是比较健康的布局状态驱动方式，后续优先延续，不建议退回“直接写死元素宽度”的命令式方案。

### 4.3 移动端 / 窄屏

当前系统已经提供响应式降级：

- 中屏改为双列 + 分层布局
- 小屏改为单列堆叠

即使以后视觉大改，也建议保留这一思想：  
先保持结构语义，再调整版式，而不是移动端重新做一套产品逻辑。

## 5. 核心区域语义

### 5.1 左侧地点栏

职责：

- 展示当前位置的主视觉
- 展示地点名、地点简介
- 承载任务清单

关键实现：

- 图片容器：`.location-art`
- 文字覆盖：`.location-copy`
- 任务列表：`.flow-panel`

规则：

- 左上图不是装饰图，而是“当前位置识别锚点”
- 图像必须带遮罩，确保文字可读
- 任务列表使用竖线 + 小方框的低噪声结构，不用卡片化列表

### 5.2 代码区

职责：

- 编辑脚本
- 运行控制
- 展示静态诊断
- 提供轻量编辑器交互

结构：

- 顶部标签：`.code-label`
- 控制条：`.controls`
- 编辑器框：`.editor-frame`
- 行号：`.line-numbers`
- 高亮层：`.script-highlight`
- 输入层：`textarea`
- 诊断区：`.diagnostics-panel`
- 自动补全：`.autocomplete`

规则：

- 代码编辑采用 `textarea + highlight overlay`，不使用 `contenteditable`
- 行号和代码首行必须共用同一顶部内边距
- 行号显示两位数，如 `01`
- 可点击跳转、自动补全、错误跳转都属于“轻量 IDE 行为”，可以增强，但不要演化成重量级 IDE

### 5.3 地图演出区

职责：

- 展示真实世界状态
- 承载挂机态与剧情态
- 提供拖拽、缩放、运镜和实体动画

结构：

- 视口：`.canvas-viewport`
- 世界层：`.canvas-world`
- 有限可达区：`.playfield-barrier`
- 地图实体层：`.grid`
- 剧情覆盖层：`.story-dialogue`
- 运行时中断：`.runtime-toast`

规则：

1. 地图从启动开始就是真实地图  
   不存在假图或过渡用占位地图。

2. 网格与实体同层变换  
   拖拽或缩放时，网格不能留在原地，实体必须和世界坐标一起运动。

3. 世界无限，游玩区域有限  
   画布允许无限舞台感；可行动区域通过 barrier 表达。

4. 剧情是覆盖层，不是切场  
   演出模式暂停挂机内容，但不卸载底层地图。

5. 中断信息靠近舞台  
   运行时阻塞、碰撞、越界、开销耗尽等中断，应该在地图区内反馈，并驱动自动暂停。

### 5.4 右侧系统栏

职责：

- 展示资源
- 展示模块 / 状态
- 挂载设置
- 收纳研发调试信息

结构：

- 统计区：`.terminal-section`
- 计量条：`.meter-stack`
- 设置抽屉：`.settings-panel`
- DevLog 抽屉：`.dev-panel`
- 操作按钮：`.action-stack`

规则：

- 玩家可见资源与研发日志必须分层
- 存档属于设置项，不属于常驻主信息
- 未解锁地点不应出现在这里，而应属于大地图层

## 6. 组件模式

### 6.1 按钮

统一按钮语义：

- 默认：透明底 + 细边框
- Hover：边框和文字提亮
- Active：亮琥珀底 + 反黑字
- Toggle：用 `data-active` 或 `aria-pressed`

按钮不是装饰块，而是终端控制元件，所以尽量保持扁平、窄高、低圆角。

### 6.2 面板抬头

所有抬头共享同一语法：

- 紧凑横条
- 左右对齐
- 低饱和标题
- 局部琥珀高亮字段

适用于：

- 左栏标题
- 地图标题
- 右栏标题
- 代码区标签

### 6.3 任务项

组成：

- 左侧小复选框
- 左边 2px 装饰竖线
- 文本主体
- 当前项高亮竖线

这是一种“流程引导组件”，不是待办软件风格的 checkbox list。

### 6.4 诊断列表

静态诊断位于代码区底部，表现为：

- 错误 / 警告两级
- 左侧色条
- 可点击跳转到对应行

后续扩展时保持以下边界：

- 编译期问题进这里
- 运行期中断不要塞进这里当主反馈

### 6.5 自动补全

语义：

- 面向轻量脚本作者的候选词提示
- 支持键盘上下选择、Tab / Enter 接受
- 支持按 token / segment 匹配

规则：

- 定位必须动态计算
- 面板宽度固定在紧凑范围
- 信息只给“词条 + 类型 + 简短提示”

### 6.6 运行时中断提示

语义：

- 循环过程中出现阻塞时的即时反馈
- 与自动暂停机制绑定

规则：

- 必须本地化
- 与编译错误分离
- 视觉上比普通日志更强，但比整屏弹窗更轻

### 6.7 剧情对话框

语义：

- 地图底部的剧情演出层
- 支持逐页推进
- 平时隐藏

规则：

- 无对话时完全隐藏
- 有对话时停挂机，不销毁底图
- 正文使用正文级字体，不使用纯终端等宽字体硬顶

## 7. 交互系统

### 7.1 播放控制

当前玩家控制语义已经稳定：

- `Play`
  - 等价于部署并开始运行
- `Pause / Resume`
  - 由同一个播放按钮承担状态切换
- `Frame`
  - 快进一帧
- `Speed`
  - 在 `x1 / x5 / x10` 等倍率间切换
- `Stop`
  - 停止并重置到未运行状态

后续新增运行控制时，应保持这一套心智模型，不要重新引入“部署”和“播放”两个并列主按钮。

### 7.2 地图交互

- 支持拖拽
- 支持滚轮缩放
- 首次加载有轻微推荐镜头运镜
- 运镜期间禁用玩家拖拽与缩放
- 拖拽和缩放都有边界约束

后续若增加大地图或地点切换，也建议沿用“相机状态对象统一管理”的方式，而不是让多个组件各自改 `transform`。

### 7.3 故事与挂机切换

- 挂机模式：世界持续可运行
- 演出模式：暂停挂机，底部弹出对话
- 结束对话后：回到挂机模式

这是一条核心产品规则，后续即使加入更多剧情演出，也建议保留这层模式切换语义。

### 7.4 本地化

当前本地化原则已经比较明确：

- 所有玩家可见文本进入 `apps/web/i18n.csv`
- 以稳定 key 查字典
- 默认语言跟随系统语言
- 支持中文、英文切换

后续不要再往 `main.js` 塞大块硬编码玩家文案。

## 8. 信息层级

可以把当前系统理解为四层：

1. 世界层
   - 地图、实体、位移、障碍、资源
2. 操作层
   - 代码、播放、步进、速度、停止
3. 状态层
   - 任务、资源、模块、设置
4. 调试层
   - 诊断、Diff、DevLog、运行时中断原因

扩展时优先问一个问题：  
这个新信息属于哪一层？  
如果层级判断错了，界面就会重新变吵。

## 9. 扩展规则

### 9.1 可以稳定复用的部分

- 四栏工作台结构
- 终端式深色底 + 琥珀焦点色
- 标题 / 控制条 / 内容区的分层关系
- 任务列表的竖线语法
- 编辑器的 overlay 结构
- 世界层统一变换的地图相机
- 剧情覆盖真实地图的模式
- 调试信息降权显示
- CSV 本地化字典方案

### 9.2 允许演化的部分

- 实体图形具体造型
- 地点主视觉素材
- 大地图视图
- 模块面板内容
- 设置抽屉内容
- DevLog 呈现密度
- 运行时中断的具体视觉样式

### 9.3 不建议轻易打破的部分

- `TapeScript` 与游戏层的边界
- 玩家文本与程序逻辑分离
- 编译诊断与运行时中断分离
- 网格与实体统一世界坐标
- 折叠侧栏让中心工作区扩张

## 11. 给后续扩展的建议

### 11.1 如果要做新的玩家界面

优先复用：

- 色板
- 面板分层
- 标题条
- 控制按钮语义
- 世界层 / 覆盖层 / 调试层的分层逻辑

### 11.2 如果要做研发工具

比如未来的图形编辑器、关卡编辑器、脚本调试器：

- 可以沿用同一套终端壳层
- 允许提高信息密度
- 允许暴露更多调试信息
- 但不应反向污染玩家主界面的安静层级

### 11.3 如果要迁移到桌面壳

建议保持：

- 同一套数据和本地化来源
- 同一套视觉令牌
- 同一套交互状态机

也就是说，壳层可以换，设计语言和信息架构尽量不要重做。

## 12. 一句话总结

当前这套设计系统，本质上是一套“终端式脚本工作台 + 可演出的真实地图舞台”的混合界面语言。它最重要的不是某个按钮长什么样，而是始终把玩家主路径、研发辅助信息、真实世界状态和脚本控制面板放在各自正确的层里。

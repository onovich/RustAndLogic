# Rust & Logic 阶段设计总表

本文把主线阶段拆成统一结构，供关卡、机制、数值和脚本系统对齐使用。

字段说明：

- `阶段目标`：这一阶段要让玩家学会什么
- `主编程主题`：该阶段唯一最重要的编程问题
- `新增 API / 能力`：玩家在这一阶段获得的新表达能力
- `旧脚本失效点`：为什么玩家必须重构
- `关卡 / 系统考试点`：这一阶段如何检验玩家是否学会

## 1. 主线阶段矩阵

| 阶段 | 阶段目标 | 主编程主题 | 新增 API / 能力 | 旧脚本失效点 | 关卡 / 系统考试点 |
| --- | --- | --- | --- | --- | --- |
| M1 唤醒机器人 | 建立最基础动作反馈与情感连接 | 顺序执行 | `Move` `Turn` `PickUp` `Drop` | 还没有旧脚本，重点是确认“命令真的生效” | 写出第一段 3~5 行脚本并实际移动 / 拾取 |
| M2 稳定拾荒循环 | 从单步动作进入可持续工作 | 循环 + 条件判断 | `Check()` `Has()` `If` `IfNot` `Then` 背包 / 电量读数 | 只会写固定动作序列的脚本无法持续工作 | 自动出门拾荒、装满回家、充电后继续 |
| M3 临时据点 | 从单机器人进入小型生产系统 | 状态切换 + 函数拆分 | `Unload` 基础合成 基础修理 充电站交互 | “捡到东西就一直捡”的脚本会断在背包、电量、耐久上 | 自动卸货、回充、维修、继续工作 |
| M4 危险垃圾区 | 第一次明确引入风险收益判断 | 风险评估 + 回撤逻辑 | 辐射 / 噪音 / 耐久感知 更高价值资源扫描 | “看见资源就捡”会因为损坏、断电、辐射而失效 | 在危险区稳定搬回高价值材料且不频繁报废 |
| M5 黑市与工坊 | 从固定阈值刷资源升级到动态调度 | 动态成本判断 | `Trade` `Craft` `Unlock` `GetCost` 风格接口 | 写死数值阈值的脚本会在升级成本变化后抖动或断供 | 自动根据下一个升级缺口切换资源目标 |
| M6 角斗场试炼 | 建立战斗 AI 主线 | 状态机 + 目标优先级 | `Attack` `Block` `Dash` `Repair` `GetEnemyDistance` `GetEnemyHP` | 拾荒逻辑无法处理实时敌我状态 | 对不同敌人写出对应战斗策略并稳定取胜 |
| M7 废墟迷宫 | 建立探索与寻路系统 | 记忆 + 构图 + 找路 | `CanMove` `ScanDirection` `MarkTile` `RememberPosition` `FindPath` | 按列遍历、贴着资源走的脚本不再适用 | 通过浅层到中层迷宫，找到并带回目标物 |
| M8 旧世界设施 | 把剧情奖励转成系统能力 | 系统整合 | 路径规划增强 战术预测 持久记忆 更高级扫描 | 单一场景脚本无法覆盖探索、战斗、经济混合要求 | 用新 API 打开被锁死的设施与剧情线 |
| M9 多机器人小队 | 从单体自动化进入全局调度 | 并发 + 任务分配 + 共享状态 | `SpawnRobot` `AssignTask` `Broadcast` `SharedMemory` `TaskQueue` | 复制单机脚本到多机上会造成冲突、抢资源、空转 | 多台机器人分工协作完成据点 / 战斗 / 探索目标 |
| M10 深层迷宫 / 联赛 | 综合检验中后期系统 | 调度整合 + 高级策略 | 高级战斗模块 高级路径模块 高级交易模块 | 局部最优脚本会在复杂目标竞争中失效 | 同时维持生产、战斗、探索三条线 |
| M11 旧核心觉醒 | 终局压力测试 | 综合自动化 + 异常恢复 | 核心设施控制 更高阶集群能力 | 人工介入多、脆弱脚本会在长周期压力下崩掉 | 持续作战、补给、修理、进攻一体化 |
| M12 自主机械聚落 | 建立长期优化目标 | 完整运营系统 | 聚落级任务、全局策略层 | 只会处理单任务的脚本无法扩张 | 从零启动、自主扩张、排行榜挑战 |

## 2. 阶段设计硬规则

每个新阶段都必须满足以下至少四条：

1. 有唯一主编程主题
2. 有新增表达能力，而不是只有数值奖励
3. 有明确的旧脚本失效点
4. 有新的失败方式
5. 有新的资源约束
6. 有一场明确的“考试关”

如果一个阶段只有新资源，没有新编程问题，它不应进入主线。

## 3. 旧脚本失效点清单

为了便于策划对照，下面把每次重构的原因单独列出：

- M1 -> M2：固定动作序列无法持续工作
- M2 -> M3：只会拾荒的脚本不会卸货、维修、回充
- M3 -> M4：稳定循环缺少风险判断，会在危险区报废
- M4 -> M5：固定阈值无法应对动态成本与交易系统
- M5 -> M6：经济脚本无法处理实时战斗决策
- M6 -> M7：战斗状态机无法解决空间寻路问题
- M7 -> M8：单场景探索脚本无法处理剧情设施的复合要求
- M8 -> M9：单机器人脚本无法处理并发与共享资源
- M9 -> M10：局部任务脚本无法做全局调度
- M10 -> M11：无异常恢复的自动化无法承受终局压力
- M11 -> M12：单轮通关逻辑无法支持长期自扩张系统

## 4. API 解锁分层建议

### 基础动作层

- `Move`
- `Turn`
- `PickUp`
- `Drop`
- `Wait`

### 基础判断层

- `Check()`
- `Has()`
- `Is()`
- `If`
- `IfNot`
- `Then`

### 生存循环层

- `Unload`
- 电量、背包、耐久读取
- 维修与充电相关接口

### 风险与经济层

- 区域风险读取
- `Trade`
- `Craft`
- `Unlock`
- `GetCost`

### 战斗层

- `Attack`
- `Block`
- `Dash`
- `Repair`
- 敌我状态读取

### 探索层

- `CanMove`
- `ScanDirection`
- `MarkTile`
- `RememberPosition`
- `FindPath`

### 集群层

- `SpawnRobot`
- `AssignTask`
- `Broadcast`
- `SharedMemory`
- `TaskQueue`

## 5. 使用方式

后续做以下工作时，优先查这张总表：

- 新关卡设计
- 新模块设计
- 新 API 解锁设计
- 新资源投放设计
- 教学节奏设计
- 玩家脚本能力成长设计

它的用途是回答一个核心问题：

> 当前阶段到底在教玩家什么，  
> 为什么玩家必须重写代码？

# Rust & Logic M1~M4 系统需求

本文把 [docs/milestone-1-4-playable-spec.md](/D:/LabProjects/RustAndLogic/docs/milestone-1-4-playable-spec.md) 转成可执行的系统需求，用于指导前四个里程碑的实现切片。

它只覆盖首个可玩闭环，不覆盖中后期黑市、角斗场、迷宫、多机器人等内容。

## 1. 文档目标

M1~M4 的目标不是“把世界做大”，而是先把下面这条成长路径做扎实：

1. 玩家写出第一段会驱动机器人行动的脚本。
2. 玩家写出第一套稳定的自动拾荒循环。
3. 玩家把“机器人单体行为”升级成“基地运转工作流”。
4. 玩家第一次因为危险区规则而被迫重构脚本。

这四步完成后，玩家应该明显感受到：

> 自己不是在手操一个单位，  
> 而是在维护一个会自己运行的小系统。

## 2. 当前实现基线

当前仓库里已经存在一套最小原型，不是从零开始：

### 2.1 TapeScript 现状

- 已有独立于 UI 的编译器、VM、硬件边界。
- 已支持：
  - 标签与 `Goto`
  - 条件前缀 `IfTrue` / `IfFalse`
  - 动作：`Move` `MoveToward(Home)` `Turn` `PickUp` `Drop` `Unload` `Fire` `Wait` `Repair`
  - 查询：`Check(...).Has(...)` `Is(...)` `IsEmpty()` `Any()` `IsFull()` `Below()` `Above()`
- 已有编译错误、标签解析、watchdog 故障等基础能力。

### 2.2 game-sim 现状

- 已有小尺寸确定性网格世界。
- 已有机器人坐标、方向、血量、护甲、武器、货舱。
- 已有基础资源、障碍、掉落物、基地。
- 已有序列化快照、diff、基础日志。
- 尚未形成真正的：
  - 电量循环
  - 耐久/危险区循环
  - 基地设施循环
  - 分阶段解锁内容

### 2.3 Web UI 现状

- 已有可运行的网页工作台。
- 已有代码编辑、地图演出、任务/资源侧栏、本地化、剧情层、运行控制。
- 已能展示编译诊断与运行时中断。
- 目前还没有与 M1~M4 深度绑定的“教学任务、阶段脚本要求、基地设施反馈、危险区反馈”。

因此，M1~M4 的工作重点不是发明新壳层，而是把现有原型收束成真正的早期游戏闭环。

## 3. 顶层边界要求

### 3.1 TapeScript 域

TapeScript 只负责：

- 语法
- 编译
- VM 执行
- 条件标志与跳转
- 通过硬件边界表达“查询”和“动作”

TapeScript 不负责：

- 地图渲染
- 剧情
- 存档表现
- 经济数值
- UI 教学流程
- 基地面板或模块面板

### 3.2 Rust & Logic 游戏域

游戏层负责：

- 世界规则
- 资源、设施、危险区、任务、进度
- 机器人硬件状态
- 快照生成
- UI 呈现与反馈

游戏层通过显式的编译器 / VM / hardware boundary 调用 TapeScript，而不是把脚本规则散落到 UI 中。

## 4. M1~M4 玩家可见需求

### 4.1 M1 唤醒机器人

玩家必须能够：

- 看到机器人在地图上移动、转向、拾取、放下。
- 写出 3~5 行脚本并立即看到结果。
- 理解“代码一行 -> 世界一个动作反馈”。

玩家不需要：

- 复杂资源链
- 危险区
- 战斗
- 多机器人

### 4.2 M2 稳定拾荒循环

玩家必须能够：

- 判断前方是否有资源、是否空、是否到家、货舱是否已满。
- 写出“出门 -> 拾取 -> 装满回家 -> 卸货 -> 再出门”的循环。
- 感知电量或等价生存成本，理解为什么机器人不能永远无脑运行。

### 4.3 M3 临时据点

玩家必须能够：

- 明确知道基地不只是交付点，而是工作流枢纽。
- 感知回家后的几个动作差异：卸货、补给、维修、合成。
- 开始把脚本组织成几个明确职责段落。

### 4.4 M4 危险垃圾区

玩家必须能够：

- 明确感知危险区与普通区的收益差异。
- 因风险、耐久、电量或其他消耗而遭遇旧脚本失效。
- 通过新判断逻辑稳定从危险区带回更高价值资源。

## 5. 系统需求拆解

## 5.1 关卡与世界需求

M1~M4 需要一套能分阶段解锁的世界规则，而不是一次性塞入所有机制。

必须具备：

- 阶段化地图或场景配置：
  - M1：极小安全区
  - M2：基础拾荒区
  - M3：带基地设施的工作区
  - M4：普通区 + 危险区
- 同一套坐标/快照模型可复用到所有阶段。
- 每个阶段有清晰的任务目标和验收标准。
- 危险区必须在地图上可识别，而不是纯隐藏规则。

不要求：

- 大地图旅行
- 未解锁地点分支
- 复杂事件系统

## 5.2 资源与经济需求

M1~M4 只需要早期资源层，不需要完整中后期经济。

必须具备：

- 基础资源：
  - `Scrap`
  - `Battery` 或等价早期能源物料
  - 1~2 种用于首条生产链的补充材料
- 明确区分：
  - 地图掉落物
  - 机器人货舱
  - 基地库存
- 卸货后资源应进入基地库存，而不是始终挂在机器人身上。
- 至少一条短配方，证明“回收 -> 入库 -> 合成 -> 消耗/升级”成立。
- 危险区产出应明显高于普通区，且最好引入新资源，而不是只增加数量。

不要求：

- 动态市场价格
- 黑市贸易
- 多层工业链

## 5.3 机器人硬件与生存循环需求

M2 开始就要建立“机器人不能无限白跑”的约束。

必须具备：

- 货舱容量
- 电量或等价行动成本
- HP / 耐久中的至少一条稳定损耗链
- 在基地恢复的机制：
  - 卸货
  - 充能/补给
  - 维修
- 危险区额外损耗：
  - 更快掉耐久
  - 或辐射/腐蚀等环境值

表现要求：

- 玩家能读懂“为什么这次循环停了”。
- 失败原因要能区分是：
  - 货满
  - 电量不足
  - 受损过高
  - 越界/阻挡
  - 危险值过高

## 5.4 基地与设施需求

M3 必须把基地从“出生点”提升成“系统节点”。

必须具备：

- 家/基地的明确地图身份。
- 至少三类基地交互能力中的两类有实际玩法价值：
  - 卸货
  - 充能
  - 维修
  - 合成
- 设施规则进入模拟层，而不是写死在 UI 文案里。
- 设施是否可用、需要消耗什么、产出什么，应能从快照中读出。

不要求：

- 复杂设施摆放编辑
- 多房间基地建设

## 5.5 TapeScript 语言需求

前四个里程碑不应该再大改语法范式，重点是把当前语法表面稳定下来。

必须具备：

- 支撑 M1~M4 的最小指令面：
  - 动作
  - 查询
  - 跳转
  - 条件前缀
- 编译器能给出面向初学者的错误信息：
  - 未知指令
  - 参数非法
  - 标签缺失
  - 指令槽超限
- 运行时能稳定区分：
  - 正常暂停在物理动作后
  - 程序结束
  - watchdog 故障
  - 硬件动作失败

阶段性解锁建议：

- M1：只教动作
- M2：开始教 `Check` + `IfTrue` / `IfFalse`
- M3：不一定新增语法，但需要让玩家开始组织职责段
- M4：通过新查询值支撑风险判断

不要求：

- 函数定义
- 复杂数据结构
- 多值返回
- 用户自定义枚举系统

## 5.6 模拟层需求

模拟层必须成为 M1~M4 的真实权威，而不是 UI 自己拼行为。

必须具备：

- 每步执行后的确定性快照。
- 查询与动作都只经由硬件边界。
- 可以按阶段配置：
  - 地图尺寸
  - 掉落物
  - 基地位置
  - 障碍
  - 危险区
  - 设施规则
- 运行日志里能输出足够解释性的事件。
- 失败事件能被 UI 直接消费并给玩家反馈。

额外要求：

- 需要给 UI 足够的数据做阶段任务判定。
- 需要给调试层足够的数据判断“脚本为什么卡住”。

## 5.7 UI 与教学需求

Web UI 不只是个调试器，要开始承担“早期教学壳层”职责。

必须具备：

- 阶段任务清单与完成状态。
- 默认示例脚本与阶段对应。
- 编译诊断与运行时中断分离。
- 对资源、货舱、电量、耐久、基地状态有清楚反馈。
- 播放、暂停、单帧、加速这些控制与阶段任务协同工作。
- 剧情/引导只负责解释任务，不替代真实模拟。

编辑器侧至少应支持：

- 两位数行号
- 基础语法高亮
- 错误列表点击跳转
- 轻量候选词提示

## 5.8 存档与进度需求

M1~M4 不需要完整 meta progression，但要有最小进度承载方式。

必须具备：

- 保存当前场景状态
- 保存当前脚本
- 保存当前阶段进度
- 恢复后仍能继续验证同一任务

不要求：

- 多周目结构
- 跨地点世界状态图谱

## 5.9 日志、诊断与遥测需求

为了支撑这四个里程碑的教学，日志不是附属品。

必须具备：

- 编译期诊断
- 运行时中断原因
- 最近几步行为日志
- 任务完成或失败时的明确提示

推荐具备：

- 区分“动作失败”和“逻辑故障”
- 区分“玩家脚本有问题”和“环境条件不满足”

## 6. 按里程碑列出的最小验收

### M1 验收

- 玩家能写出并运行第一段动作脚本。
- 机器人能清楚表现移动、转向、拾取、放下。
- UI 能显示编译成功、失败、当前步骤日志。

### M2 验收

- 玩家能做出稳定的自动拾荒循环。
- 货舱与电量约束真实影响脚本。
- “满载回家”与“低电回家”都能在模拟层成立。

### M3 验收

- 基地至少承担卸货与补给两类玩法职责。
- 至少一条短生产链成立。
- 玩家需要明确写出回家后的处理逻辑，而不是只会来回走。

### M4 验收

- 普通区脚本直接拿去危险区会明显失效。
- 危险区收益高于普通区。
- 玩家通过新增判断逻辑后，能稳定回收危险区产物。

## 7. 本阶段非目标

前四个里程碑明确不做：

- 黑市价格波动
- 角斗场正式战斗循环
- 迷宫探索系统
- 多机器人调度
- 大规模世界旅行
- 复杂剧情分支
- 完整桌面壳/发布系统

## 8. 对后续实现的约束

后续所有实现切片都应优先回答这三个问题：

1. 这个改动属于 TapeScript 还是游戏层？
2. 它是否服务于 M1~M4 的真实可玩闭环？
3. 它引入的是新编程问题，还是只是在堆新资源？

如果一个改动答不上第 2 或第 3 条，就不应挤进当前阶段。

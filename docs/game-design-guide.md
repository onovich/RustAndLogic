# Rust & Logic 游戏设计指南

本文基于以下两份设计研究文档整理而成：

- `designs/编程农场设计文档.pdf`
- `designs/废土挂机编程游戏设计拆解文档.pdf`

目标不是复述原文，而是提炼出适合 `Rust & Logic` 的长期设计准则，供关卡、机制、数值、系统和后续策划评审统一使用。

## 1. 核心定位

`Rust & Logic` 的本质不是“废土题材挂机游戏”，也不是“单纯的编程教学游戏”。

它更准确的定位应该是：

- 废土机器人自动化 RPG
- 废土机器人操作系统模拟器

玩家的成长路径不是：

- 从弱角色练成强角色

而是：

- 从写几条动作指令
- 到写一个稳定的自动化脚本
- 再到写一个能自我生存、自我升级、自我战斗、自我探索的系统

因此，游戏的长期爽点不应建立在“数值越来越大”上，而应建立在：

- 代码越来越聪明
- 自动化越来越完整
- 系统越来越自洽
- 玩家越来越像在经营一个机器人文明

## 2. 从竞品提炼出的核心方法

《编程农场》最值得继承的不是题材，也不是某个具体机制，而是它的内容填充公式：

> 不要用“更多资源”填充内容，  
> 要用“会让旧代码失效的新规则”填充内容。

它的优秀之处在于，每个阶段都引入新的编程问题，而不是单纯加一种新作物。

其背后的设计模式可以抽象为：

`新内容 = 新资源 + 新约束 + 新 API + 新失败方式 + 新代码结构`

例如：

- 新资源：高价值材料
- 新约束：辐射、限电、路径阻塞、敌人
- 新 API：扫描、测量、记忆、路径查询、战斗状态读取
- 新失败：卡死、损坏、迷路、断供、战败
- 新代码结构：循环、条件、状态机、优先级调度、路径规划、并发调度

这条公式应该成为 `Rust & Logic` 的内容生产总则。

## 3. 我们的设计总纲

### 3.1 三条成长线

游戏必须同时推进三条线，并在关键节点交汇：

1. 机器人身份线
   - 破损废品
   - 可修复工具
   - 旧世界设备
   - 核心设施钥匙
   - 新机械聚落核心

2. 玩家能力线
   - 单步命令
   - 循环
   - 条件判断
   - 状态机
   - 自动经济系统
   - 战斗 AI
   - 迷宫探索系统
   - 多机器人调度
   - 完整自动化框架

3. 世界探索线
   - 垃圾场边缘
   - 危险废区
   - 黑市与工坊
   - 角斗场
   - 废墟迷宫
   - 深层设施
   - 城市核心
   - 机械聚落

### 3.2 四类核心体验

整个游戏的主体验应由四类循环组成：

1. 拾荒循环
   - 看见资源
   - 过去
   - 捡起
   - 装满回家
   - 卸货
   - 重复

2. 据点循环
   - 回充
   - 维修
   - 合成
   - 升级
   - 再部署

3. 战斗循环
   - 识别敌人
   - 评估距离 / 血量 / 热量 / 冷却
   - 决定攻击 / 防御 / 后撤 / 修理

4. 探索循环
   - 扫描
   - 标记
   - 记忆
   - 找路
   - 决定冒险 / 回撤

### 3.3 设计目标

玩家在中后期应该明显感受到：

- 我不是在“按按钮”
- 我不是在“手操机器人”
- 我是在维护一个会自己工作的系统

## 4. 关卡设计原则

### 4.1 一关只教一个主问题

每个新区域 / 新阶段都应有一个明确的“主编程问题”。

例如：

- 垃圾场边缘：循环与基础拾荒
- 稳定垃圾场：回家、卸货、充电
- 危险垃圾区：风险判断与回撤
- 黑市 / 工坊：动态成本与经济调度
- 角斗场：状态机与目标优先级
- 迷宫：寻路与记忆
- 多机器人：并发与任务分配

避免一上来同时引入三种新问题，否则玩家不会重构，只会混乱。

### 4.2 新关卡必须破坏旧脚本

关卡设计要主动制造“旧脚本不再可靠”的时刻。

例如：

- 普通区：看到资源就捡
- 危险区：看到资源先评估值不值得冒险
- 战斗区：不能只向前走，要读敌人状态
- 迷宫区：不能只按列扫描，要构图找路

如果新关卡只是让旧脚本继续工作但效率更高，那它不算成功扩展，只是线性数值膨胀。

### 4.3 每关同时给“新能力”和“新考试”

理想的阶段结构：

1. 给一个新 API / 新模块
2. 提供一个低风险场景让玩家尝试
3. 再给一个会让旧逻辑失效的复杂场景
4. 最后用一场综合关卡检验玩家是否真的学会

也就是说：

- 教学关
- 练习关
- 考试关
- 综合关

这比“每张图只是更难”要健康得多。

### 4.4 关卡目标要以“系统行为”描述，而不是以“过图”描述

优先用这样的目标：

- 写出能持续回收 Scrap 的稳定循环
- 写出能自动回家充电并继续工作的脚本
- 写出能判断辐射风险并主动撤离的脚本
- 写出能在角斗场里持续击败某类敌人的状态机

而不是只用：

- 收集 100 Scrap
- 打赢 3 场战斗

前者更贴合本作的系统感。

## 5. 机制设计原则

### 5.1 模块升级 = 表达能力升级

模块升级最好优先解锁新 API，而不是只加数值。

例如：

- 扫描器升级 -> `scan_area()`
- 电池模块 -> `get_battery()`
- 耐久模块 -> `get_durability()`
- 导航核心 -> `remember_position()` / `find_path()`
- 战斗模块 -> `get_enemy_distance()` / `get_enemy_hp()`
- 集群协议 -> `spawn_robot()` / `assign_task()`

如果升级只给数值，玩家会觉得自己是在玩普通 RPG。
如果升级能改变能写什么代码，玩家才会觉得自己在“造系统”。

### 5.2 流动资源是条件，不是库存

以下资源应被视为“策略条件”，而不是普通库存：

- Energy
- Heat
- Durability
- Noise
- Radiation

它们的作用不是让玩家囤，而是让玩家写判断。

正确的问题是：

- 什么时候该撤退？
- 什么时候该停火？
- 什么时候该回家修理？

而不是：

- 怎么把 Heat 堆到 999

### 5.3 危险区必须提供“高收益 + 高不确定性”

危险区如果只是“更难但更慢”，玩家不会喜欢。
危险区要满足：

- 单次收益显著更高
- 但需要额外逻辑才能稳定赚到

例如：

- 辐射区掉高级芯片
- 酸雨区掉稀有液体
- 旧电厂掉能源核心

但同时：

- 耐久掉得快
- 传感器有误报
- 会吸引敌人
- 电量消耗变快

这样新逻辑才值得写。

### 5.4 角斗场不是战斗支线，而是战斗 AI 主线

角斗场的职责：

- 教状态机
- 教敌人识别
- 教优先级
- 教冷却与热量管理
- 教实时决策

它不应该只是额外刷材料的副本。
它是“资源调度系统”之外的第二套核心考试场。

### 5.5 迷宫不是宝箱系统，而是寻路系统

迷宫的重点不应是“有多少奖励”，而应是：

- 墙和门
- 陷阱和能量限制
- 标记与记忆
- 回溯与风险收益
- 路线规划

迷宫和角斗场的编程主题必须区分：

- 角斗场考实时决策
- 迷宫考空间认知与路径规划

### 5.6 多机器人不是产量倍增器，而是调度题

多机器人阶段的重点不是“产出翻倍”，而是：

- 谁去哪里
- 谁先拿资源
- 谁回基地维修
- 谁探索
- 谁留守
- 谁执行交易 / 战斗 / 运输

如果多机器人只是把单机逻辑复制四份，就浪费掉了最有价值的中后期设计空间。

## 6. 数值设计原则

### 6.1 早期回报要快

前 30 分钟的数值目标主要服务于“验证自动化是值得的”。

必须满足：

1. 单次反馈快
2. 升级成本低
3. 玩家能明显看到脚本比手动强

否则玩家还没进入系统乐趣，就先被前期资源墙劝退。

### 6.2 高级目标必须依赖低级资源链

不要让高级资源独立生产。

要形成类似这样的依赖：

- 高级芯片 <- 危险区产物
- 危险区探索 <- 装甲 / 能源 / 维修
- 装甲 <- Scrap + Plate
- Plate <- 熔炉
- 熔炉 <- 基础供能系统

这样玩家就必须维护整条链，而不是只盯一个数字。

### 6.3 用滞回区间代替单阈值

单阈值容易抖动：

- Scrap < 1000 就去刷
- Scrap >= 1000 就停

这会导致逻辑频繁来回切。

应该改成双阈值：

- 低于低线，进入恢复模式
- 高于高线，退出恢复模式

例如：

- 电量 < 20% -> 回家充电
- 电量 > 80% -> 恢复工作

### 6.4 对流动资源使用节流，而不是追满

像 `Energy` / `Heat` / `Noise` 这类值，不适合做“永远追到满”的库存目标。

更好的策略是：

- 低于危险线才处理
- 回到安全线就退出
- 只在升级或关键任务需要时短时补充

这样脚本才会稳定，不会出现：

- 一直充电
- 一直散热
- 一直刷某种临时资源

### 6.5 风险收益要足以支付代码复杂度

如果一个新机制让玩家多写三倍代码，却只多 5% 收益，玩家只会觉得累。

因此新阶段的收益提升要明显到：

- 能看出“值得为它重构”
- 但又不能高到“无脑跳过旧系统”

建议遵循：

- 安全区：稳定、低收益、低维护
- 危险区：高收益、高维护、高波动

### 6.6 数值瓶颈应有层次

一个阶段里，最好有：

1. 表层目标资源
2. 中间材料
3. 消耗品
4. 基础供给

例如：

- 表层目标：Arena Token
- 中间材料：Weapon Part
- 消耗品：Ammo / Heat Sink
- 基础供给：Scrap / Wire / Cell

这能自然形成“短期、 中期、长期”三个层次的运营判断。

## 7. 内容填充公式

后续新增区域、敌人、工坊或模式时，统一先过这张表：

1. 新资源是什么？
2. 新约束是什么？
3. 新 API 是什么？
4. 新失败方式是什么？
5. 它迫使玩家写出什么新代码结构？
6. 它会让哪一种旧脚本失效？

如果一个新内容回答不出第 5、6 条，就说明它大概率只是“换皮内容”，不够资格进入主线。

## 8. 对当前项目的落地方向

结合项目现状，当前最值得优先落的不是终局系统，而是以下四个策划层目标：

1. 先把“单机器人稳定自动拾荒循环”做扎实
2. 再把“据点回充 / 维修 / 卸货 / 合成”的最小生产链做出来
3. 再引入第一个真正会让旧脚本失效的危险规则
4. 再开启黑市 / 角斗场 / 迷宫三个中期大分支

这意味着短期不要急着堆：

- 过多敌人种类
- 过多稀有资源
- 过多剧情设施

而应该优先把“每个阶段到底教什么编程问题”钉死。

## 9. 结论

`Rust & Logic` 最应该坚持的设计原则只有一句：

> 玩家不是在操控机器人，  
> 玩家是在把机器人训练成一个系统。

因此：

- 关卡要制造新的编程问题
- 机制要推动代码重构
- 数值要制造隐性瓶颈和调度压力
- 奖励要优先解锁表达能力

只要这四条不丢，这个项目就不会沦为“编程农场换皮”，而会真正长成一个有题材、有情感、有系统深度、有长期优化空间的废土自动化 RPG。

# Rust & Logic M1~M4 实现 Backlog

本文把 [docs/m1-m4-system-requirements.md](/D:/LabProjects/RustAndLogic/docs/m1-m4-system-requirements.md) 拆成实现 backlog，并强制按顶层域分开：

- `TapeScript`
- `game-sim`
- `web UI`

目标不是列一个无限长清单，而是产出“能指导下一轮切片实现”的顺序化 backlog。

## 1. 当前基线判断

当前仓库不是白纸状态，因此 backlog 要分成三类：

- `已有，可直接沿用`
- `已有，需要扩展`
- `全新需求`

### 1.1 TapeScript 基线

已具备：

- 编译器
- VM
- 条件查询与跳转
- 适合前四个里程碑的最小动作集

当前更像“技术原型已成立”，下一步重点是：

- 稳定早期指令面
- 补齐早期教学所需的错误反馈
- 明确哪些查询值由游戏层提供

### 1.2 game-sim 基线

已具备：

- 网格、机器人、掉落物、基地、障碍
- 快照、diff、基础执行日志

当前缺口主要在：

- 货舱与基地库存的边界
- 电量/耐久循环
- 设施与危险区
- 阶段任务所需的可观测数据

### 1.3 Web UI 基线

已具备：

- 编辑器、地图、资源栏、剧情层、运行控制、本地化

当前缺口主要在：

- 与 M1~M4 任务绑定
- 对设施/危险区/中断原因的清楚反馈
- 默认脚本、任务目标、阶段切换的教学包装

## 2. 实现优先级总览

### P0

先把“单机器人稳定自动拾荒 + 基地补给闭环”做出来。

### P1

再把“基地设施 + 首条生产链 + 危险区重构点”做出来。

### P2

最后补足教学、日志、任务验收、内容配置化，形成可持续扩展的前四里程碑框架。

### TapeScript 说明

这一层当前不建议大改语法，也不建议引入函数定义、复杂枚举、复杂链式调用。  
M1~M4 的价值在于把“现有简单语法”打磨成真正可教、可用、可扩展的起跑线。

## 7. 建议执行顺序

推荐不要按“先语言、再模拟、再 UI”三大块整片推进，而是按垂直切片推进：

1. `P0-1`  
   明确基地库存/货舱边界 + UI 可见化 + 对应任务文案。
2. `P0-2`  
   引入电量循环 + 回家补给闭环 + 对应运行时反馈。
3. `P0-3`  
   稳定 M2 教学任务，验证“自动拾荒循环”真的成立。
4. `P1-1`  
   加入基地设施中的充能/维修。
5. `P1-2`  
   加入首条短生产链。
6. `P1-3`  
   引入危险区、高价值资源、风险查询。
7. `P1-4`  
   完成 M4 教学任务与危险区重构验证。
8. `P2`  
   做场景配置化、任务验收事件、编辑器体验打磨。

## 8. 下一轮最值得立刻实现的切片

如果只选最值得立刻做的一轮，我建议是：

1. 在 `game-sim` 中把“基地库存 vs 货舱”正式落地。
2. 同时引入最小电量循环。
3. 在 `web UI` 中把资源栏与任务列表改成围绕这两个变化展示。
4. 用一个 M2 smoke 场景验证“自动拾荒 -> 回家卸货 -> 补给 -> 再出门”。

原因很简单：  
这一步一旦站住，M1 是它的简化版，M3 是它的扩展版，M4 则是在这条稳定循环上加风险约束。

## 9. 判定一个 backlog 项是否值得进入当前阶段

当前只要同时满足下面三条，就值得进 M1~M4：

1. 它强化了“写脚本驱动机器人”的主体验。
2. 它会创造新的编程问题，而不是只加一个新资源名。
3. 它能落在 `TapeScript`、`game-sim`、`web UI` 之一的明确边界内。

如果不满足这三条，优先级就应该后移。

## 设计师数值规格参考

用户后续补充了一版更偏工程化的参数表，可作为像素级复刻时的数值参考基线。使用原则如下：

- 当参考图里有清晰视觉事实时，以参考图优先。
- 当参考图不够细、但参数表给了明确字号、边距、颜色、描边、面板宽度、按钮状态时，可直接采用参数表。
- 当参数表与现有设计图或已确认交互冲突时，以设计图和用户明确口径优先，再把参数表当作近似建议。

当前已确认可直接吸收的高价值参数包括：

- 全局终端基线：
  - `Courier New` / monospace
  - 12px 基准字号
  - 大写终端视觉语言
  - 全屏扫描线覆盖层
- 顶部彩虹条：
  - 高 3px
  - 亮度/对比增强
  - 带暗色外发光
- 左侧地点 Hero：
  - 图上文字需依赖深色渐变遮罩保证可读性
  - 地点标题、状态、副文案之间要维持明显字号层级
- 通用按钮：
  - 透明底 + 细描边为默认态
  - 主按钮为琥珀底反黑字
  - 激活态允许使用内发光
- 编辑器与系统栏：
  - 候选词、诊断面板、系统字段、细进度条都应使用更紧凑的终端密度，而不是偏卡片化样式

当前仍需谨慎使用、不能机械照搬的参数包括：

- 地图绝对定位 `top/left` 与固定 2000px 画布坐标：当前项目使用的是可拖拽缩放的相机状态，不应回退成纯静态 DOM 画布定位方案。
- 运行时错误提示在地图顶部居中：这与当前更接近参考图的左下角 halt toast 口径冲突，暂不直接采用。
- 全局 `text-transform: uppercase`：对玩家界面可作为视觉语言参考，但代码编辑器与脚本文本本体仍应避免被纯视觉大小写变换污染。

### 最新补充参数（pass11 已吸收口径）

这一轮又补了一组更精确的滤镜、辉光、遮罩和动效参数。当前项目已经吸收、且适合继续沿用的部分如下：

  - 底部暗晕使用 `0 0 10px rgba(0,0,0,0.5)`
- 全屏扫描线：
  - 采用 `linear-gradient(rgba(13,8,5,0) 50%, rgba(0,0,0,0.15) 50%)`
  - 条纹密度 `background-size: 100% 4px`
  - 覆盖层透明度 `0.5`

  - 统一口径为 `rgba(242, 141, 53, 0.4)`
  - 主要用于机器人实体、对话标签、进度条填充、候选词高亮等高能量焦点
- 对话框与浮层：
  - 剧情对话框使用 `backdrop-filter: blur(4px)`
  - 自动补全面板阴影使用 `0 4px 12px rgba(0,0,0,0.6)`
  - 运行时 halt toast 阴影使用 `0 0 12px rgba(192,57,43,0.3)`
- 动效口径：
  - 面板宽度变化继续使用 `0.3s cubic-bezier(0.4, 0, 0.2, 1)`
  - 进度条宽度过渡使用 `0.3s`
  - 当前地图仍保留项目既有的相机模型与 intro 运镜，不回退成设计草稿里的固定 `top/left` DOM 拖拽方案

这一组参数的适用原则：

- 只把它当作像素级还原的数值基线，不把它误当成新的产品交互规则。
- 如果它和参考图、当前确认过的交互逻辑冲突，仍以参考图和项目现有交互为准。
- 如果将来大地图、更多演出层、更多调试浮层出现，优先复用这组辉光/遮罩口径，避免重新发明另一套视觉语法。

## 实现风险备注（来自设计师 AI，需按当前架构筛选）

以下内容来自设计稿产出侧的实现规避建议。它们不是当前项目的强制事实，也不代表现有代码已经存在对应问题。这里将其作为后续 UI 重构与交互落地时的风险清单使用，并结合当前工程实现标注适用性、用途与项目判断。

### 1. DOM ID 重复

- 原建议：避免同一页面出现重复的 DOM `id`，尤其是运行时提示节点。
- 适用性：中等适用
- 用途：用于约束后续地图提示、运行时浮层、调试面板等单实例节点的挂载方式，避免查询命中错误节点。
- 当前项目判断：当前 Web UI 没有设计稿里提到的 `error-toast` 双节点问题，但后续若增加地图角落错误提示、剧情浮层、状态浮层，仍应坚持单实例 `id` 或改用组件内 scoped query。

### 2. 地图拖拽首次移动出现 NaN

- 原建议：若通过 `style.left/top` 配合 `parseInt` 做拖拽，初始空字符串会导致首次拖拽跳位或报错。
- 适用性：当前不适用
- 用途：可作为未来若改回 DOM 绝对定位拖拽方案时的排错提示。
- 当前项目判断：当前地图画布使用 `canvasState.x / y / scale` 驱动变换，不依赖 `style.left/top` 解析，因此不存在该设计稿中的首次拖拽 NaN 风险。

### 3. `contenteditable` 会破坏语法高亮结构

- 原建议：不要在带语法高亮内嵌 `span` 的 DOM 上直接使用 `contenteditable` 作为代码编辑器。
- 适用性：当前不适用，但应保留为架构警告
- 用途：用于约束代码编辑器架构，避免未来重构时回退到脆弱的 DOM 原地编辑方案。
- 当前项目判断：当前编辑器已采用 `textarea + highlight overlay` 路线，和该建议一致；后续若考虑更复杂编辑能力，应继续沿用 overlay 或引入成熟编辑器，而不是回到 `contenteditable`。

### 4. 自动补全定位不能写死

- 原建议：候选词下拉的位置应根据当前光标、行高、容器坐标动态计算，不能使用固定像素值。
- 适用性：高适用
- 用途：约束编辑器自动补全、悬浮提示、跳转浮层等 UI 的定位实现方式。
- 当前项目判断：当前项目已经动态计算补全菜单位置，这条建议应保留为持续约束，避免后续版式大改时又出现硬编码坐标。

### 5. 碰撞箱、边界和地图尺寸硬编码耦合

- 原建议：不要把机器人尺寸、障碍尺寸、边界阈值、地图尺寸等数字散落在 CSS 和脚本多个位置。
- 适用性：高适用
- 用途：用于指导地图运行时、镜头边界、碰撞系统和表现层插值共用一套几何参数来源。
- 当前项目判断：虽然设计稿里的具体数字与当前项目不一致，但“几何常量集中管理”对我们当前的地图、拖拽缩放、障碍边界、角色移动表现都直接重要，属于值得优先遵守的工程规则。

### 6. 折叠面板的 `!important` 陷阱

- 原建议：折叠态宽度不应靠 `!important` 强压，否则后续 JS tween、拖拽调宽或状态切换会难以接管。
- 适用性：中等适用
- 用途：用于约束左右侧栏折叠、展开、动画和将来可调整宽度的实现方式。
- 当前项目判断：当前侧栏主要通过 `body` 数据属性切换布局，不是设计稿中的 `.collapsed { width: ... !important; }` 方案，但该建议对未来继续演进布局依然有参考价值。

### 7. 对话分页点应 scoped query

- 原建议：对话分页点、对话按钮等节点查询不要用全局选择器污染多个组件实例。
- 适用性：中等适用
- 用途：用于约束剧情演出模式、多地点剧情组件、未来教程弹窗等多实例 UI 的查询范围。
- 当前项目判断：当前对话框仍偏单实例，但以后若同屏出现多个对话、引导卡片或剧情层，scoped query 会明显更稳，应优先采用。

### 8. 不要依赖全局 `event`

- 原建议：事件处理函数不要直接读浏览器注入的全局 `event`，而应显式接收参数。
- 适用性：当前不适用
- 用途：属于通用浏览器兼容与可维护性规则。
- 当前项目判断：当前项目使用 `addEventListener` 并显式接收事件对象，没有设计稿里的 `onclick=\"runCode()\"` 风险，因此本条保留为一般性规约即可。

### 9. 按钮需要显式绑定清单

- 原建议：播放、步进、倍速、停止、设置、存档等按钮在视觉稿里存在时，程序侧必须逐一核对事件绑定与状态互斥逻辑。
- 适用性：中等适用
- 用途：作为 UI 大改后的交互联调检查表，防止样式改完但行为没接上。
- 当前项目判断：当前主要运行控制按钮已经有绑定，不存在设计稿草图里“只有 HTML 没有事件”的问题；但以后如果完全换新版式，这条建议仍适合作为联调 checklist。

### 10. 运行时错误与编译诊断要分通道

- 原建议：编译诊断和运行时中断不要混在同一状态数组里，它们的生命周期和展示位置不同。
- 适用性：高适用
- 用途：指导代码区诊断、地图区运行时错误提示、自动暂停原因说明等通道分层。
- 当前项目判断：这条对当前架构非常重要。我们已经区分“编辑器编译诊断”和“运行时自动暂停/阻塞”，后续如果补充地图左下角报错提示、日志、Toast 或对话式报错，也应继续保持两条状态链路分离。

### 11. 地图拖拽与点击交互冲突

- 原建议：地图上的可点击元素需要有统一的“不可触发拖拽”标识，否则点击会误触画布拖拽。
- 适用性：中等适用
- 用途：用于约束无限画布上的热点、地点入口、剧情节点、浮层按钮、地图工具条等未来交互元素。
- 当前项目判断：当前地图交互还相对集中，但随着大地图、地点切换、剧情入口和演出区 UI 继续增多，统一的 `.no-drag` 或等价机制会很有帮助。

### 12. `text-transform: uppercase` 的视觉欺骗

- 原建议：如果界面把文本强制显示为大写，而代码内容实际保留原样，可能导致视觉上看到的和解析器读取的不一致。
- 适用性：低适用
- 用途：提醒视觉层大小写变换不要误作用到代码编辑体验与语言语义。
- 当前项目判断：当前项目的代码编辑器本身不是全局大写显示，但样式表中部分按钮和标题使用了大写视觉语言。因此这条建议不属于当前已发生问题，不过在处理代码区、诊断区、候选词下拉时，仍需避免对脚本文本本体施加纯视觉大小写变换。

### 13. 面板 `gap` 带来的缝隙副作用

- 原建议：如果布局依赖容器 `gap` 作为分隔线，未来动态增删面板时容易出现不想要的缝隙或沉浸感中断。
- 适用性：低到中等适用
- 用途：用于指导四栏布局、调试面板、全屏演出态、可折叠面板之间的分隔策略。
- 当前项目判断：当前布局并非完全照搬设计稿，但这条提醒对后续“演出模式 / 挂机模式”切换、全屏地图态、隐藏调试区态仍有参考意义。需要沉浸式连续背景时，优先显式控制边框与分隔线，而不是完全依赖 `gap`。

## 建议使用方式

- 当后续开始落地新版 UI 时，先把本节作为“实现前检查表”过一遍。
- 若发现设计稿建议与当前工程实现冲突，以当前项目架构边界为准，再决定是否吸收其思路。
- 若未来再收到新的设计侧实现建议，也应继续以“适用性 + 用途 + 当前项目判断”的形式追加到本节，避免把草图细节直接误当成项目事实。
